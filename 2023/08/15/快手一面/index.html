typeof null输出是什么？为什么？

使用 typeof 操作符检测 null 的数据类型会返回 “object”。

在 JavaScript 最初的实现中，用 32 位的机器码来表示值的类型信息，对象的类型信息被固定为 0x00000010，而 null 的值被定义为全零的指针（null pointer），对应的类型信息也是全零，即 0x00000000。由于 null 的类型信息是全零，因此被错误地判断为 “object” 类型。

instanceof的检测机制是什么？

首先，它会获取对象的原型链。然后，它会遍历对象的原型链，逐个比较每个原型对象与指定的构造函数的原型对象是否相等。如果找到匹配的原型对象，则返回 true，表示对象是该构造函数（或者其原型链上）的实例。如果遍历完整个原型链都没有找到匹配的原型对象，则返回 false，表示对象不是该构造函数（或者其原型链上）的实例。

对this的理解

this 关键字用于引用当前执行代码的上下文对象。

this 的值是在函数调用时动态确定的，取决于函数的调用方式。this 的值可能是以下几种情况之一：

  1. 全局上下文：在全局作用域中（在任何函数之外）访问 this，它会指向全局对象（在浏览器环境中通常是 window 对象）。
  2. 函数中的 this：在函数内部使用 this，默认情况下，它会指向调用函数的对象。但是，如何调用函数决定了函数中 this 的值。
    • 方法调用：当函数作为对象的方法调用时，this 将指向该对象。
    • 构造函数调用：当函数作为构造函数调用时（使用 new 关键字创建对象），this 将指向新创建的对象。
    • 函数调用：当函数作为普通函数调用时，this 将指向全局对象（在严格模式下是 undefined）。
    • 箭头函数：箭头函数中的 this 是词法上绑定的，它的值取决于箭头函数的定义环境，而不是函数调用的方式。
  3. 使用 call() 或 apply() 显式指定 this：可以使用 call() 或 apply() 方法来手动指定函数中的 this 值。这两个方法将第一个参数作为 this 值传递给函数。

如何理解闭包？举个例子？

闭包是指有权访问另一个函数作用域中变量的函数。简单来说，闭包是由函数和其相关的引用环境组合而成的。

JavaScript 中的闭包可以通过内部函数访问外部函数的变量，即使外部函数已经执行完毕。它可以保留对这些变量的引用，使得外部函数中的变量在内部函数中仍然可访问和使用。

通常，闭包可以通过以下几种方式创建：

  1. 函数内部定义函数：内部函数可以访问外部函数的变量。
  2. 返回内部函数：将内部函数作为外部函数的返回值，在外部函数执行完成后，返回的内部函数仍然可以访问外部函数的变量。
  3. 事件监听器：将函数作为事件监听器，事件触发时，函数仍然可以访问事件绑定时的作用域。

闭包的主要特点包括：

  1. 保留外部函数作用域：内部函数可以访问和使用外部函数的变量。
  2. 长期存储变量：外部函数的变量可以被保存在内部函数中，即使外部函数已经执行完毕。
  3. 防止变量污染：内部函数可以使用外部函数的变量，而不必将变量声明为全局变量，从而避免了全局命名冲突和变量污染。

通过闭包，我们可以实现一些特定的功能，例如封装私有变量、实现模块化等。但是，使用闭包时应该避免内存泄漏问题，即及时释放不再使用的闭包。

对模块化开发的理解？

前端模块化开发是将前端代码按照功能和逻辑进行拆分，每个模块独立开发、测试和维护的一种开发方式。模块化开发有助于增强代码的可维护性、可复用性和可拓展性，同时也能提高开发效率。

传统的前端开发方式中，所有的代码都写在一个文件中，导致文件臃肿、难以维护、容易产生命名冲突和依赖管理问题。而模块化开发则能够解决这些问题，它将代码划分为多个模块，每个模块负责特定的功能。每个模块有自己的作用域和依赖关系，模块之间通过导入和导出的方式进行通信。

常见的前端模块化开发方案有以下几种：

  1. CommonJS：Node.js 采用的模块化规范，使用 require 导入模块，使用 module.exports 或 exports 导出模块。
  2. AMD（Asynchronous Module Definition）：异步模块定义，RequireJS 使用的模块化规范，支持异步加载模块，使用 define 定义和导入模块。
  3. ES6 模块化：ES6（ECMAScript 2015）引入的官方模块化规范，使用 import 导入模块，使用 export 或 export default 导出模块。
  4. UMD（Universal Module Definition）：通用模块定义，可以兼容 CommonJS 和 AMD 的写法，适用于跨平台使用的模块化规范。

通过使用模块化开发，我们可以将功能相对独立的代码拆分成多个模块，减少代码的耦合度，提高代码的可维护性和复用性。同时，模块化开发也为代码的拓展和团队协作提供了良好的基础。

Ajax的实现原理是什么？

Ajax（Asynchronous JavaScript and XML）的实现原理是通过浏览器提供的XMLHttpRequest对象（现在一般使用fetch API）与服务器进行异步通信。

具体的实现过程如下：

  1. 创建XMLHttpRequest对象或使用fetch API，它是封装了HTTP请求的对象，可以实现在后台与服务器进行通信。对于老的浏览器不支持XMLHttpRequest对象，则需要创建ActiveXObject对象（IE浏览器）。
  2. 使用XMLHttpRequest对象（或fetch API）的open方法，设置HTTP请求的一些参数，如请求的方式（GET、POST等）、请求的URL地址等。
  3. 可选地，设置请求的头部信息，如Content-Type等。
  4. 使用XMLHttpRequest对象（或fetch API）的send方法发送请求，发送的参数可以是字符串或FormData对象，用于传递给服务器。
  5. 服务器接收到请求后，处理请求并返回相应的数据。
  6. 当服务器返回响应后，XMLHttpRequest对象（或fetch API）会触发相应的回调函数，通过监听onreadystatechange事件（或使用Promise处理fetch API的响应）来获取服务器返回的数据。
  7. 在回调函数中可以对返回的数据进行处理，如更新页面内容、更新数据等。

通过使用Ajax，可以实现页面的局部刷新而无需重新加载整个页面，提高用户体验。同时，由于是异步通信，页面的其他操作不会被阻塞。

数组有哪些方法？

  1. push(item1, item2, …): 将一个或多个元素添加到数组的末尾，并返回新的长度。
  2. pop(): 删除数组的最后一个元素，并返回被删除的元素。
  3. shift(): 删除数组的第一个元素，并返回被删除的元素。
  4. unshift(item1, item2, …): 在数组的开头添加一个或多个元素，并返回新的长度。
  5. concat(arr1, arr2, …): 连接两个或多个数组，并返回一个新的数组。
  6. slice(start, end): 从原数组中截取(start, end]范围的元素，并返回一个新的数组。
  7. splice(start, deleteCount, item1, item2, …): 从指定位置删除或替换元素，并在该位置插入新的元素，返回一个包含被删除元素的数组。
  8. reverse(): 颠倒数组中元素的顺序，并返回新的数组。
  9. sort(compareFunction): 对数组进行排序，默认按照Unicode码点排序，可以传入一个比较函数进行自定义排序。
  10. indexOf(item, startIndex): 返回数组中第一个找到的指定元素的索引，如果不存在则返回-1。
  11. lastIndexOf(item, startIndex): 返回数组中最后一个找到的指定元素的索引，如果不存在则返回-1。
  12. includes(item, startIndex): 判断数组是否包含指定元素，如果包含返回true，否则返回false。
  13. forEach(callback): 遍历数组中的每个元素，并执行回调函数。
  14. map(callback): 遍历数组中的每个元素，并执行回调函数，根据回调函数返回的值创建一个新的数组。
  15. filter(callback): 遍历数组中的每个元素，并执行回调函数，根据回调函数返回的值过滤出符合条件的元素，创建一个新的数组。
  16. reduce(callback, initialValue): 从左到右遍历数组中的每个元素，并执行回调函数，接收上一个回调函数的返回值和当前元素作为参数，最终返回一个累计的结果值。
  17. reduceRight(callback, initialValue): 从右到左遍历数组中的每个元素，并执行回调函数，接收上一个回调函数的返回值和当前元素作为参数，最终返回一个累计的结果值。
  18. find(callback): 遍历数组中的每个元素，并执行回调函数，返回第一个符合条件的元素，如果没有符合条件的元素则返回undefined。
  19. findIndex(callback): 遍历数组中的每个元素，并执行回调函数，返回第一个符合条件的元素的索引，如果没有符合条件的元素则返回-1。

什么是重绘和重排？

重绘是指当 DOM 元素样式改变，但是不影响其布局的情况下，浏览器会重新绘制这个元素。重绘不涉及 DOM 树的结构变化，而只是改变元素的外观。

重排是指当 DOM 结构发生改变或者元素的布局属性发生改变，浏览器会重新计算 DOM 树上的元素在页面中的位置和大小，以确定元素的新位置和尺寸。重排的开销很大，因为浏览器需要重新计算布局，并重新绘制所有受影响的元素。

重排会引起页面的重新渲染，而重绘只会重新绘制受影响的部分。因此，如果可以避免重排，可以有效提高页面性能。

应该尽量减少重排的频率，可以通过以下几个方法来减少重排的次数：

  • 使用 CSS 的 transform 属性替代 top、left 等属性来进行位移操作，因为 transform 不会触发重排；
  • 将要操作的多个 DOM 元素的样式通过添加一个 CSS 类名的方式一起修改，然后再把它们的样式一次性应用到 DOM 上，这样可以减少重排的次数；
  • 尽量在操作 DOM 之前，将需要操作的 DOM 元素从文档中隐藏，操作完成后再显示，这样可以避免频繁的重排；
  • 使用 requestAnimationFrame 方法来优化动画，因为它会在浏览器下一次重绘之前执行动画操作，可以减少重排的次数。

什么是CSS动画？

CSS动画是一种利用CSS样式规定的动画效果。它允许开发者通过定义一些规则和关键帧，来实现页面中元素的动态变化，如平移、旋转、缩放、透明度等。

CSS动画通过CSS的@keyframes规则定义一系列关键帧，指定元素在不同的时间点上的样式状态。然后使用CSS的animation属性将这些关键帧应用到元素上，指定动画的名称、持续时间、延迟时间、重复次数、播放方式等。

CSS动画具有以下优点：

  • 简单易用：只需要几行CSS代码就可以创建出漂亮的动画效果，无需编写复杂的JavaScript代码。
  • 性能高效：CSS动画在现代浏览器中会使用硬件加速，可以获得更好的性能表现。
  • 兼容性良好：CSS动画在大多数现代浏览器中都有很好的支持，并且可以通过添加浏览器前缀来扩充兼容性。

使用CSS动画时，可以通过调整关键帧之间的动画持续时间和属性变化方式，来创建出多样化的动画效果。通常配合其他的CSS属性和选择器，如transition、transform、opacity等，可以实现更加丰富的交互效果和用户体验。

浏览器打开一个页面的时候，需要启动哪些进程？

当浏览器打开一个页面时，通常会启动以下进程：

  1. 主进程（Browser Process）：这是浏览器的主要进程，负责控制浏览器窗口、处理用户输入、与操作系统进行通信等。它还负责管理其他进程，如渲染进程、插件进程等。
  2. 渲染进程（Renderer Process）：每个标签页通常都会有一个独立的渲染进程。它负责解析HTML、CSS和JavaScript，并将它们转换成可视化的页面。渲染进程还负责处理用户的交互事件、执行JavaScript代码等。
  3. GPU进程（GPU Process）：在支持GPU加速的浏览器中，会有一个独立的GPU进程，它与主进程和渲染进程进行通信，负责处理与图形相关的任务，如页面的渲染、图形绘制等。
  4. 插件进程（Plugin Process）：如果页面使用了插件（如Flash、Java等），则会为插件启动一个独立的进程，专门用于加载和运行插件。

除了以上的进程，还有一些其他的进程可能被启动，具体取决于浏览器的功能和配置。例如，浏览器可能会启动网络进程、媒体进程等来处理与网络请求和媒体播放相关的任务。

对Chrome渲染机制的理解？

Chrome的渲染机制主要涉及以下几个步骤：

  1. 解析HTML：浏览器接收到服务器返回的HTML文件后，首先会进行HTML解析，将HTML代码转换成一棵DOM树（Document Object Model）。在解析过程中，浏览器会根据HTML标记构建DOM节点，并确定节点之间的关系。
  2. 解析CSS：接下来，浏览器会解析CSS样式文件，并将样式规则应用于DOM树中的各个节点，形成带有样式的DOM。
  3. 构建渲染树：浏览器根据DOM树和CSS样式信息构建渲染树（Render Tree）。渲染树只包含需要显示的节点，并且每个节点都与对应的CSS样式进行关联。
  4. 布局计算：渲染树构建完成后，浏览器需要对渲染树进行布局计算，确定每个节点在页面中的位置和大小。这个过程称为布局或回流（Layout or Reflow）。
  5. 绘制：在布局计算完成后，浏览器会遍历渲染树，将每个节点转换成绘制指令，即将节点的内容绘制到屏幕上。
  6. 栅格化与合成：为了提高性能和效果，Chrome会将页面分成多个小块叫做图块（Tiles），每个图块都会独立进行栅格化和合成。栅格化是将绘制指令转换成位图，而合成是将多个图块的位图合成成最终的页面。

上述渲染过程是在渲染进程（Renderer Process）中进行的。同时，Chrome在多进程架构下还有其他进程参与渲染相关任务，例如合成进程（Compositor Process）用于图块的合成，GPU进程用于图形处理等。

常用的React Hooks？

  1. useState：用于在函数组件中引入状态变量，并返回一个当前状态以及更新状态的函数。
  2. useEffect：用于在函数组件中执行副作用操作，例如订阅/取消订阅事件、发送网络请求、手动操作DOM等。
  3. useContext：用于在函数组件中访问上下文（Context）中的值。
  4. useReducer：用于在函数组件中使用Reducer模式来管理复杂的状态逻辑。
  5. useRef：用于在函数组件中创建一个可持久化的引用，类似于class组件中的ref。

  8. useImperativeHandle：用于在使用ref时自定义暴露给父组件的实例值，类似于class组件中暴露给父组件的实例方法。

对React.memo()的理解？

React.memo()是React提供的一个高阶组件，用于对函数组件进行性能优化。

React.memo()接受一个函数组件作为参数，并返回一个经过优化的组件。它通过对组件的传入 props 进行浅比较来确定是否重新渲染组件。如果传入的 props 没有发生变化，那么 React.memo() 将会使用上一次的渲染结果，避免不必要的重新渲染。

使用React.memo()可以帮助我们避免在父组件重新渲染时，无需重新渲染的子组件也被重新渲染，从而提高应用程序的性能。特别是在组件树的层级较深，或者组件的渲染逻辑较复杂时，使用React.memo()可以有效减少不必要的渲染开销。

在上述示例中，父组件ParentComponent包含一个计数器和一个子组件ChildComponent。使用React.memo()对ChildComponent进行了优化。

当父组件中的计数器发生变化时，只有父组件会重新渲染，而子组件ChildComponent则不会重新渲染。这是因为React.memo()对子组件的props进行了浅比较，并决定了在相同的props情况下，是否复用上一次的渲染结果。

需要注意的是，React.memo()默认只进行浅比较，如果组件接收到一个复杂对象或引用类型的props，那么可能会导致意外的重新渲染。为了避免这种情况，可以使用自定义的props比较函数作为第二个参数传递给React.memo()，实现更精确的比较逻辑。

对React Fiber的理解？

React Fiber是React 16及更高版本中的一种新的协调机制，它改变了React的渲染机制，使得React可以实现更灵活的调度和处理方式，从而提升应用的性能和响应能力。

传统的React调度方式是使用递归调用，当组件进行更新时，React会从根组件开始递归遍历整个组件树，计算出新的Virtual DOM树，然后与旧的Virtual DOM进行比较，最后再进行渲染。这种方式有时会导致长时间的主线程阻塞，造成用户界面的卡顿。

React Fiber引入了一种新的调度算法，将更新的任务拆分成多个小任务，通过任务优先级的概念进行调度，从而可以在每一帧中划分出一小部分任务执行，然后将执行权交还给浏览器，使得界面保持响应。在下一帧开始时，React Fiber会根据剩余的未完成任务，选择适当的任务优先级继续执行。

React Fiber的引入带来了以下几个优势：

  1. 异步渲染：React Fiber通过拆分任务和任务优先级调度，实现了异步渲染。它可以将任务分解成多个小片段，根据优先级和渲染帧频来适应不同的设备和网速，提供更好的用户体验。
  2. 更细粒度的控制：Fiber架构将任务拆解成一个个的工作单元，每个工作单元都有自己的优先级。这样可以在每一个工作单元之间进行中断和恢复，同时也可以中断和恢复整个渲染过程，使得React可以更细粒度地控制任务的执行顺序和优先级。
  3. 支持增量渲染：由于React Fiber可以将任务分成多个片段，它可以在一次渲染过程中进行增量渲染，即根据用户视口的变化，先渲染可见部分的内容，然后渐进式地渲染剩余的内容。这样可以提升首次加载的速度，减少页面的加载时间。

React17更新了哪些内容？

React 17 是一个属于 “Silent Migration（静默迁移）” 的版本，即在这个版本中，没有引入新的功能或者语法，主要是对现有 API 进行了调整和改进，主要包括以下几点更新：

  1. 事件处理程序默认以冒泡模式注册：在 React 17 之前，在使用 React 的事件处理时，事件处理程序默认以捕获模式注册。React 17 将事件处理程序的默认注册模式改为了冒泡模式，与浏览器原生事件的工作模式更加一致。
  2. 移除了废弃的生命周期方法：React 17 移除了一些已经被标记为 “废弃” 的生命周期方法，如 componentWillMount、componentWillReceiveProps和 componentWillUpdate。这些方法并没有被删除，而是标记为 UNSAFE_ 前缀。
  3. 单元测试改进：React 17 对于单元测试提供了更好的支持，添加了新的工具和 API来简化对 React 组件的测试，如 render 函数、act 工具和 createEvent 函数。
  4. 不再需要隐式的 React 导入：React 17 在编译阶段不再需要隐式的导入 import React from 'react'，这意味着可以在组件中省略对 React 的导入，但需要保留对 React DOM 的导入。

Webpack的构建流程？

Webpack 的构建流程可以概括为以下几个步骤：

  1. 配置：首先，需要创建一个 webpack 的配置文件，一般命名为 webpack.config.js。在配置文件中，指定入口文件、输出文件的路径、以及其他的一些配置选项，如 loaders、plugins 等。
  2. 解析依赖：Webpack 会根据配置文件中指定的入口文件，查找所有相关的依赖文件。它会从入口文件开始递归地解析依赖树，找到所有需要被打包的文件。
  3. 加载资源：在解析依赖的过程中，对于不同类型的文件，Webpack 会使用相应的 loader 进行加载和转换。Loader 可以将非 JavaScript 文件（如 CSS、图片等）转换成 Webpack 可以识别和处理的模块。
  4. 模块打包：Webpack 会将所有的模块打包成一组静态资源。它会根据入口文件和依赖关系，构建一个输出的依赖图，确定模块的加载顺序。
  5. 输出：Webpack 会根据配置文件中指定的输出路径和文件名，将打包生成的静态资源文件输出到指定的目录中。一般情况下，Webpack 会将多个模块打包成一个或多个 bundle 文件，可通过配置文件中的 splitChunks 选项来拆分打包结果。
  6. 优化：Webpack 还包含一些优化功能，如代码压缩、文件合并、使用缓存等。这些优化操作可以在配置文件中进行配置。
  7. 开发和生产环境的区分：Webpack 可以根据不同的环境进行不同的配置，例如在开发环境中可以启用 HMR（热模块替换），而在生产环境中可以开启代码压缩等优化。

Webpack常见的plugin有哪些？

  1. HtmlWebpackPlugin：用于自动生成 HTML 文件，并将打包生成的资源文件（如 JavaScript、CSS）自动引入到 HTML 文件中。
  2. CleanWebpackPlugin：用于每次构建前清理输出目录中的文件。
  3. MiniCssExtractPlugin：将 CSS 文件从打包生成的 JavaScript 文件中分离出来，生成独立的 CSS 文件。
  4. DefinePlugin：定义全局常量，可以在代码中访问这些常量。
  5. CopyWebpackPlugin：将指定的文件或文件夹复制到输出目录中。
  6. HotModuleReplacementPlugin：启用热模块替换（Hot Module Replacement），用于在开发环境中实现页面无刷新更新。
  7. UglifyJsPlugin：用于压缩 JavaScript 代码。
  8. OptimizeCSSAssetsPlugin：用于压缩 CSS 代码。
  9. ProvidePlugin：在代码中自动加载模块，而无需 require/import。
  10. BundleAnalyzerPlugin：分析打包生成的文件的大小，帮助进行优化。

Webpack常见的loader有哪些？

  1. babel-loader：用于将 ES6+ 语法编译为 ES5，以便在旧版本的浏览器中运行。
  2. css-loader：加载 CSS 文件，并解析其中的 import 和 url()。
  3. style-loader：将 CSS 代码注入到 HTML 文件中的<head>标签中。
  4. sass-loader：加载 Sass/SCSS 文件，并将其转换为 CSS 代码。
  5. less-loader：加载 Less 文件，并将其转换为 CSS 代码。
  6. file-loader：加载文件，并在输出目录中输出文件的路径。
  7. url-loader：类似于 file-loader，但可以根据文件大小将文件转换为 base64 编码，减少网络请求。
  8. image-webpack-loader：优化图片文件，包括压缩、缩放等操作。
  9. json-loader：加载 JSON 文件，将其解析为 JavaScript 对象。
  10. xml-loader：加载 XML 文件。
  11. csv-loader：加载 CSV 文件。

Http和Https的区别？

  1. 安全性: HTTP 是明文传输，数据不加密，容易被窃听和篡改，而 HTTPS 使用 SSL/TLS 加密数据传输，保证数据的安全性。
  2. 默认端口: HTTP 默认使用端口80，而 HTTPS 默认使用端口443。
  3. 证书: HTTPS 使用数字证书对网站进行身份验证和数据加密，以防止恶意攻击者冒充网站。HTTP 不需要证书。
  4. 协议层: HTTPS 在传输层（TLS/SSL）上加密数据，而 HTTP 在应用层传输数据。
  5. 运行速度: 由于 HTTPS 需要进行加密解密过程，会消耗一些额外的计算资源，所以在一些条件较差的网络环境下，HTTPS 的传输速度可能会比 HTTP 慢一些。

常见的Http请求方法有哪些？

  1. GET：用于从服务器获取资源。GET 请求可以在 URL 中传递参数，将参数附加在 URL 的末尾以查询字符串的形式。
  2. POST：用于向服务器提交数据，请求服务器处理数据。POST 请求将数据作为请求体的一部分，而不是将数据附加在 URL 上。
  3. PUT：用于向服务器上传或更新资源。PUT 请求将资源的新版本发送给服务器，如果资源不存在，则创建一个新资源。
  4. DELETE：用于删除服务器上的资源。
  5. HEAD：与 GET 方法类似，但只请求获取资源的头部信息，而不获取实际的资源内容。
  6. OPTIONS：用于请求服务器返回对特定 URL 支持的请求方法。
  7. PATCH：用于对服务器上的资源进行部分更新。
  8. TRACE：用于在目的服务器端跟踪请求。
