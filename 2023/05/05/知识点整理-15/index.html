前端工程师的价值是什么？

前端工程师的价值在于他们能够将设计团队提供的视觉设计和用户体验转化为可执行的代码，并且能够确保该代码可靠、高效地运行在不同的浏览器和设备上。除此之外，前端工程师还负责构建优秀的用户界面，使用户能够简单、直观地与Web应用程序或网站进行交互，从而提高用户满意度和忠诚度。与此同时，前端工程师还可以通过不断学习和实践来推动Web前端技术的进步和创新，促进移动端和Web技术的跨界融合，从而驱动整个互联网行业的发展。

什么是中间人攻击？

中间人攻击（Man-In-The-Middle Attack/Attack, MITM Attack/Attack）是指攻击者与通讯双方的通讯过程中进行拦截和篡改，从而使双方认为自己正与对方直接通信，但实际上所有传输数据都经过了攻击者的篡改和窃取。

中间人攻击可能会在通讯过程中通过伪造数字证书等方式获得通讯双方的信任。一旦攻击者实现了中间人攻击，他就可以窃取或篡改双方通信的数据，读取双方机密信息如账号密码、身份证号、银行卡信息等。

中间人攻击的常见情景包括：公共Wi-Fi网络上的登录、HTTP网站传输数据、SMTP、POP、IMAP等协议等。因此，通常建议用户在使用互联网时使用加密连接方式，如HTTPS、VPN等，并保持软件和系统的更新，以减少中间人攻击的风险。

中间人攻击的流程？

  1. 监听：攻击者在通信的中间位置部署监听设备，如Wi-Fi网络嗅探器等，以截获双方的通信数据包。
  2. 伪造：攻击者会伪装成一个可信的通讯双方参与通信，从而使双方相信自己是真正的通信对象。
  3. 拦截：攻击者会拦截双方通信过程中传输的数据包，修改或替换其中的数据内容，并将伪造的数据重新发送给对方，使得双方都不知道数据包已被篡改。
  4. 窃取：攻击者利用拦截到的通信数据包，获取其中的私密信息，如账号密码、银行卡信息等。
  5. 篡改：攻击者还可以篡改数据包中的信息，比如把转账金额改大、修改登录密码，从而掌控用户的账号和密码。

中间人攻击会涉及到公钥吗？

中间人攻击有可能涉及到公钥。在加密通信中，公钥加密方式通常使用服务器公钥加密来保护通信的安全性。中间人攻击者可以替换服务器公钥发送给客户端，使得客户端的数据其实是被攻击者拦截和篡改过的。因此，公钥的安全性是中间人攻击的一个重要因素。为了减少中间人攻击的风险，可采用一些防御措施，比如使用HTTPS协议、数字证书等。

SSLSniff攻击：

攻击者在网关截获SSL会话，替换服务器公钥证书，将公钥PKey换成自己的公钥PKey，欺骗客户端。客户端使用PKey加密信息并发送会话，中间人用私钥Skey解密客户端返回会话，从而劫持会话。同时，中间人用PKey加密明文会话并返回服务器。

过程如下：

  1. Attacker截获了客户端的say hello，可以把publicKey_attacker返回给客户端，取得客户端的信任，至此Attacker与客户端建立了安全连接。
  2. Attacker冒充客户端向服务器发送say hello，至此Attacker与服务器建立了安全连接。

这种攻击会存在一个问题会被感知到，就是Attacker的证书是伪造的不受信任的证书，所以客户端可以确认是否需要真的连接该服务器。

SSLStrip攻击：

这种攻击相对于攻击一复杂一点，但是也更加厉害，几乎可以在客户端无感知的情况下实施攻击，并且不需要伪造证书；简单来说就是这样：Attacker在客户端与服务器建立连接时，在Attacker与服务器之间形成HTTPS连接，而在客户端与Attacker之间形成HTTP连接，即将SSL层从原HTTPS连接中“剥离”。这样，既避免了在客户端验证证书时难以避免的弹框问题，又能够劫持HTTP明文数据，并同时保证客户端HTTP数据的传输，达到欺骗服务器与客户端的效果。

过程如下：

用户在浏览器地址栏中输入网址时，多数会采用直接输入网址的方式，而忽略了传输所采用的协议。例如，在登录gmail过程中，大多数用户会直接在地址栏中输入www.gmail.com，向Google服务器发送一个HTTP连接请求，而不是输入https：//www.gmail.com， 向服务器发送一个HTTPS连接请求。因此，用户通常接触到HTTPS的方式有两种：一种是 Web上的连接，比如当用户在gmail上输入用户名和密码后，点击的登录键，将用户的用户名和密码以HTTPS的形式POST到服务器。另一种是通过HTTP的302状态。 当客户端向gmail提出HTTP连接请求时，gmail服务器会返回一个REDIRECT网址，https：//www.google.com/accounts/ServiceLogin?service=mail...，用户端在接收到这个URL后，将页面重定位到该网页，并请求HTTPS连接。 从另外一个角度讲，用户通常是通过HTTP向服务器发起HTTPS连接的。而HTTP本身是以明文的形式对外传送，并不能保证数据的安全。因此，可以考虑通过对HTTP进行劫持，来实现对HTTPS劫持的目的。

  1. 客户端向服务器发起HTTP连接请求；
  2. 中间人MITM监听客户端与服务器的HTTP数据；
  3. 服务器返回给客户端的HTTP数据包被在客户端与服务器之间的中间人截获。中间人解析原HTTP数据包，将其中<a href=”https：//...”>替换成<a href=”http//...”>，将 Location： https：//... 替换成Location：http://..，同时记录下所修改的URL，并保存；
  4. 中间人将修改后的HTTP数据发送给客户端；
  5. 客户端向服务器发起HTTP连接请求；
  6. 中间人计算机解析客户端的HTTP连接请求，并与保存文件相比较。当发现存在有已修改过的HTTP URL时，将其替换成原HTTPS URL，并发送给服务器；
  7. 与服务器保持HTTPS连接，回到步骤3；
  8. 与客户端保持HTTP连接，回到步骤4。

中间人攻击如果篡改信息，服务器会发现吗？

如果中间人攻击者成功地篡改了通信传输的信息，服务器端通常无法直接发现这种攻击。因为服务器和攻击者之间的通信是直接的，中间人攻击者的篡改并不会被服务器端所感知。但是，服务器有可能能够通过异常检测、加强的日志分析等方式来发现一些不寻常的行为，如明显的数据篡改或存在异常的数据包传输。此外，服务器也可以启用一些数据完整性和消息认证等机制来检测数据是否被篡改，以降低中间人攻击的风险。

TCP/IP模型与OSI模型的区别与联系？

区别

  1. 层数：OSI模型将网络通信分为7层，而TCP/IP模型将网络通信分为5层。即OSI模型比TCP/IP模型多出了表示层和会话层两层。这也是OSI模型相对于TCP/IP模型而言显得较为冗余的原因之一。
  2. 技术实现的不同：OSI模型更注重于理论和学术领域的规范性，而TCP/IP模型则更加实际和直接，更注重提出实现的协议和解决问题的方法。
  3. 应用场景的不同：OSI模型更多地应用于大型企业网络和公共网络的标准化管理，而TCP/IP模型则更多地应用于广泛的计算机网络，如互联网、局域网和广域网等。

联系

  1. 都是层次结构，分层的目的都是为了使协议的设计和实现更加清晰、模块化。
  2. TCP/IP模型中的各层和OSI模型中的各层都包含特定的协议和标准，而这些协议和标准都是为了实现网络通信而设计的。
  3. 虽然OSI模型和TCP/IP模型的分层结构不同，并且各自都有自己独特的特性，但它们都是网络通信领域的重要参考模型，为网络通信的标准化和规范化做出了重要贡献。

CSS有哪些布局方式？

  1. 盒子模型布局（box model layout）：指定元素的宽度、高度、内边距和边框等属性，通过对盒子模型进行设置来实现布局。
  2. 浮动布局（float layout）：元素通过使用浮动属性，可以脱离标准布局流，浮动到页面的左侧或右侧，同时其他元素会环绕在其周围。
  3. 弹性布局（flexbox layout）：定义一个容器，容器里的项目按照一定的比例分配空间，可以实现自适应的响应式布局。
  4. 网格布局（grid layout）：将页面分成一个个容器来布局，通过定义行和列来划分布局区域，可以实现复杂的布局。
  5. 定位布局（position layout）：使用position属性，指定元素的位置，往往和浮动属性一起使用。
  6. 表格布局（table layout）：将元素当成表格和单元格来布局，可以实现类似传统网页设计的布局效果。
  7. 多列布局（multi-column layout）：将内容分成多个列来显示，可以实现报纸和杂志的页面设计效果。

CSS有哪些默认值？

    • font-size：medium（通常是16px）

    • line-height：normal（通常是字体大小的1.2倍）

  3. 颜色相关的默认值：
    • color：通常为黑色（#000）
    • background-color： 通常为透明（transparent）
  4. 布局相关的默认值：
    • display：通常为 block，但是对于 a、input，button等元素是 inline 或 inline-block

归并排序的原理是什么?

归并排序（Merge Sort）是一种基于分治思想的排序算法。其基本原理是将一个未排序的列表（数组）递归地分成两个子问题，直到问题规模足够小，然后将这些子问题的解进行合并以获得原始问题的解。

归并排序的实现步骤如下：

  1. 分：将原始问题（未排序的列表）递归地分为两个子问题，直到问题规模足够小，例如列表中只有一个元素。
  2. 治：将两个子问题的解进行合并以获得原始问题的解。

具体的合并过程如下：

  • 建立一个辅助数组，用来存储合并后的结果。
  • 将要合并的两个数组按顺序依次比较，将较小（或较大）的元素放入辅助数组中。
  • 如果其中一个数组中所有元素都已经被合并入辅助数组中，则将另一个数组中剩余的元素依次放入辅助数组。
  • 最后，将辅助数组中的元素复制回原始数组。

归并排序的时间复杂度为 O(n log n)，其中 n 表示列表中元素的个数。它的空间复杂度为 O(n)。可以看出，归并排序的递归深度是 log n，每次递归需要一个长度为 n 的辅助数组，因此总的空间复杂度为 O(n log n)。

快速排序的原理是什么？

快速排序（Quick Sort）也是一种基于分治思想的排序算法。其基本原理是选择一个元素作为基准值（通常选择第一个元素），将列表中所有小于基准值的元素移到基准值的左边，所有大于基准值的元素移到基准值的右边，然后递归地对基准值左右两边的子列表进行排序。

具体的实现步骤如下：

  1. 选择基准值：从列表中选择一个元素作为基准值（通常选择第一个元素）。
  2. 分区：将列表中小于基准值的元素放到基准值的左边，将大于基准值的元素放到基准值的右边，相等的元素可以放到任一侧（通常使用双指针法进行分区操作）。
  3. 递归排序：对基准值左右两边的子列表进行递归排序，直到子列表长度小于等于 1。

快速排序的时间复杂度为 O(n log n)，其中 n 表示列表中元素的个数。最坏情况下的时间复杂度为 O(n^2)，发生在列表已经有序，或者基准值选取不合理的情况下。它的空间复杂度为 O(log n)，因为递归调用的深度是 log n。快速排序是一种原地排序算法，因此它的空间复杂度是比较低的。

为什么快速排序要取一个随机数作为基准数？

快速排序算法的性能与基准值的选择有关。如果每次选择的基准值都是列表中的最小值或最大值，那么每次分割出的两个子列表的长度分别为 n-1 和 1，此时快速排序的时间复杂度就会退化为 O(n^2)，最坏情况发生的概率比较高。而对于随机选择基准值，可以尽可能地避免这种最坏情况的发生，从而保证了算法的性能。

随机选择基准值的优点在于可以减少最坏情况的概率。随机选择基准值可以让算法的平均时间复杂度达到 O(n log n)，可以在很大程度上提高算法的效率。此外，随机选择基准值也让攻击者很难找到攻击的空间，避免了一些安全漏洞。

快速排序再最差的情况下和冒泡排序相比谁更快？为什么？

在最坏的情况下，快速排序的时间复杂度是 O(n^2)，此时快速排序会退化成冒泡排序。但是，快速排序的交换次数要比冒泡排序少得多，因为每次交换只会交换基准值和另一个元素的位置。而冒泡排序在每一轮比较中都有可能进行元素交换，因此在最坏情况下，**冒泡排序的时间复杂度和交换次数都比快速排序更差，均为 O(n^2)**。

因此，尽管快速排序在最坏情况下的时间复杂度和冒泡排序相同，但是快速排序的交换次数要比冒泡排序少得多，相对来说更快一些。此外，在平均情况下，快速排序的时间复杂度要比冒泡排序更小，因此快速排序是一种更加高效的排序算法。

时间复杂度是怎么计算的？

计算算法的时间复杂度通常是通过分析算法中的基本操作次数来完成的，这些基本操作通常是循环、条件分支、加减乘除等简单的指令。我们针对输入规模 n，通过分析算法中所有基本操作执行的次数，得到一个关于 n 的函数 T(n)，就称之为算法的时间复杂度。

在计算时间复杂度时，通常只关注最高阶项，因为它反映了算法的增长速度。比如，对于 T(n) = 5n^2 + 3n + 2，我们只关心 n^2 这一项，而忽略其他项，因为当 n 足够大时，其他项相对于 n^2 来说会变得微不足道。

常见的时间复杂度有：

  • 常数时间复杂度 O(1)
  • 对数时间复杂度 O(logn)
  • 线性时间复杂度 O(n)
  • 线性对数时间复杂度 O(nlogn)
  • 平方时间复杂度 O(n^2)
  • 指数时间复杂度 O(2^n)

进程死锁的必要条件有哪些？

进程死锁的必要条件有以下四个：

  1. 互斥条件：至少有一个资源被进程独占，若其他进程需求该资源，则必须等待。
  2. 请求保持条件：当进程因请求资源而被阻塞时，进程已经获得的资源不会被主动释放。
  3. 不剥夺条件：进程已获得的资源，在未完成使用之前，它不能被剥夺。
  4. 环路等待条件：若干进程形成头尾相接的环路，每个进程都被下一个进程所占有需要的资源，进程之间因请求相互等待。

只有同时满足上述四个条件时，才有可能产生进程死锁。要避免进程死锁，可以通过破坏上述任意一个条件来避免，如：

  1. 破坏互斥条件：将独占资源变成共享资源，让多个进程同时占用。
  2. 破坏请求保持条件：进程因请求资源而被阻塞时，释放它已经占有的资源，等待资源的同时不保持资源。
  3. 破坏不剥夺条件：对某些资源，允许进程被剥夺已经获得的资源，等待后续重新获取。
  4. 破坏环路等待条件：按照一定的顺序获取资源，避免进程之间形成环路等待。

CSS中transition有哪些属性？

transition是CSS3中的一个动画效果属性，用于在状态改变时，使属性的改变过渡的平滑一些，以使CSS效果更加生动。

  1. transition-property：规定需要过渡的CSS属性的名称。
  2. transition-duration：规定过渡效果花费的时间。
  3. transition-timing-function：规定过渡效果的速度曲线。
  4. transition-delay：规定过渡效果何时开始。
  5. transition：简写属性，用于把上述4个属性合并起来，以便设置一组过渡效果。

CSS transform方法的属性及作用？

CSS transform方法可以让你在元素的显示方式上增加一些特效，例如旋转、缩放、平移、倾斜等。下面是CSS transform方法常用的属性及作用：

  1. translate：用于将元素在水平方向和垂直方向上移动。
  2. rotate：用来旋转元素，旋转角度为正表示顺时针旋转，负值则为逆时针旋转。
  3. scale: 用来缩放元素，可对元素进行放大或缩小操作。
  4. skew：平移元素，改变元素的形状。
  5. matrix：可以同时对元素进行多种变换，如同时进行旋转和缩放操作。
  6. perspective：用于设置元素的3D效果，可以实现透视和远近效果。

通过使用CSS transform属性，可以将不同的变换效果叠加和组合在一起，从而实现更加丰富的效果，例如3D翻转、悬浮动画、拖拽操作等，让页面更加生动和实用。

React Redux和Mobx的区别？

React Redux和Mobx都是React应用程序状态管理的库，它们有以下区别：

数据存储方式:

  • React Redux使用单一不可变的store来存储和管理应用程序中的状态。所有组件都需要从store中读取数据并且需要使用action来更新store中的数据。
  • Mobx使用可变状态的被观察对象（observable）来存储应用程序的状态。当这些被观察对象被更改时，所有的组件都会自动更新。

代码复杂度:

  • React Redux需要在组件和action之间手动传递数据和方法，因此更加显式明确，并且需要写更多的代码去连接组件和store。
  • Mobx则隐藏了这些代码的实现细节，需要更少的代码去连接组件和store，但是也需要更多的运行时代码来实现这些连接。

性能:

  • React Redux通过在store和组件之间增加了一个额外的”容器”组件（即connect函数），从而增加了一些渲染次数。
  • Mobx则通过直接引用可变状态并使用自动追踪技术避免了这些渲染次数，并且在一些场景下，性能更好。

总体来说，React Redux更加显式，需要写更多的代码去连接组件和store，但是可读性和维护性更好；Mobx则更注重自动化，需要实现更少的代码去连接组件和store，但可读性和维护性可能会受到一定的影响。选择使用哪个库应该根据具体的项目需求和团队的实际情况来决定。

jsonp实现跨域的原理？

JSONP是利用Script标签可以跨域访问的特性实现的跨域解决方案，其原理如下所示：

  1. 创建一个Script标签，其src属性指向跨域请求的URL，该URL返回的数据在JSONP回调函数中处理。
  2. 在URL中指定一个回调函数的参数名（通常为”callback”），并将回调函数的名称作为该参数的值。
  3. 返回的数据不是JSON格式的数据，而是一段Javascript代码，其中调用了该回调函数，并将数据作为该函数的参数传递。
  4. 浏览器接收到响应后会自动执行该Javascript代码，并将数据作为参数传递给回调函数。
  5. 前端定义该回调函数，根据接收到的数据进行处理。

需要注意的是，JSONP只适用于请求获取数据的场景，无法处理POST请求等其他类型的请求。此外，JSONP存在一定的安全性问题，因为回调函数是在全局作用域中执行的，如果返回的数据被恶意修改，将会产生安全问题。因此，如果需要使用跨域解决方案，建议使用CORS等更为安全的方式。

深度行为埋点是一种数据采集技术，可以跟踪并记录用户在应用程序或网站中的各种操作，如点击、滚动、滑动、输入等，以便更深入地了解用户的行为和需求，进而帮助企业优化产品和服务。

什么是BFF层？BFF层的作用，缺点与弊端？

BFF层全名为Backend for Frontend，是指在前端和后端之间新增一个层，专门用于处理前端需要的数据和接口。BFF层可以理解为是一种服务，它提供了与前端紧密相关的业务逻辑和数据处理能力。

BFF层的作用主要有以下几点：

  1. 解决前端需要多次调用后端数据接口的问题：通过BFF层可以把多个后端接口进行合并，从而减少前端调用后端接口的次数。
  2. 根据不同平台和设备提供不同的数据：BFF层可以根据前端需要的数据进行裁剪和过滤，从而提供前端所需的数据，并提升前端页面渲染速度。
  3. 提供用户认证和鉴权的接口：BFF层可以调用后端认证和鉴权的接口，从而提供用户认证和鉴权的功能。

然而，BFF层也有一些缺点和弊端：

  1. BFF层会增加系统复杂度：在系统中增加一个额外的层，会增加系统架构的复杂度和开发成本。
  2. BFF层会增加系统的运行负载：BFF层需要承担更多业务逻辑和数据处理的任务，可能会对系统资源的消耗和性能产生负面影响。
  3. BFF层需要同时兼顾多个前端应用：BFF层需要针对不同的前端应用提供不同的数据和接口，这对于开发和维护都带来一定的挑战。

对单点登录的理解与流程？

单点登录（Single Sign-On，简称SSO）是指用户只需要在一处登录，即可访问多个不同系统的资源，而无需重复输入用户名和密码。它提高了用户的使用效率，简化了用户的登录流程，同时也可以提高系统的安全性，降低了用户的密码管理负担。

单点登录的流程如下：

  1. 用户打开客户端应用（如浏览器）时，需要进行身份验证。
  2. 客户端应用将用户的凭证发送给认证服务器。
  3. 认证服务器根据用户提供的凭证验证用户是否合法，如果合法则颁发令牌。
  4. 客户端应用将令牌发送给资源服务器请求访问受保护的资源。
  5. 资源服务器收到请求后，将令牌发送给认证服务器进行验证。
  6. 认证服务器根据令牌验证该请求是否合法，如果合法则返回用户角色和权限信息给资源服务器。
  7. 资源服务器根据用户角色和权限信息来确保用户是否有权限访问请求的资源。

单点登录的实现一般可以采用以下几种方式：

  1. 使用专门的单点登录服务，并在其他系统中集成该服务。
  2. 通过采用标准的身份验证协议实现单点登录，例如OAuth、SAML、OpenID Connect等。
  3. 使用Web服务器或Web应用程序中间件来实现单点登录。

CSRF攻击的原理？如何进行防范？

CSRF（Cross-site request forgery）攻击的基本原理如下：

攻击者构造出一个恶意页面，当受害者访问该页面时，其中包含一些指向目标网站的请求，这些请求包含了攻击者植入的伪造请求参数。受害者被误导执行了这些请求后，攻击者就可以在目标网站上以受害者的身份执行一些恶意操作。因为受害者的身份已经被合法认证，所以服务器无法区分攻击者和受害者。

总之，预防 CSRF 攻击最终要做到的就是验证请求的合法性。最常见的方法是添加 token 或随机值等验证信息，在与每个请求一起发送，并在服务器端进行验证，只有验证通过才会执行用户的请求。

网络通信中端口的作用？它位于七层模型哪一层？

端口是网络通信中非常重要的概念，主要用于标识一个网络应用程序或服务。它可以帮助数据包正确地被路由到正确的程序或服务，从而保障网络通信的正常进行。换句话说，端口就是一个入口或出口，表示一个网络应用程序可以收发数据的方式。

在 TCP/IP 七层模型中，端口位于传输层（第四层）。具体来说，TCP/IP 协议栈中使用了两种主流的协议：TCP 和 UDP。TCP 协议需要使用源端口和目的端口来建立连接和进行数据传输；UDP 协议只需要使用端口即可进行数据传输。因此，通常我们说的端口主要是指 TCP/UDP 端口。

UDP 和 TCP 分别使用了一个 16 位的端口号（即 065535）来标识不同的网络应用程序，其中 01023 的端口被称为“众所周知的端口”，通常用于标识系统级别的服务（例如 SSH、HTTP、FTP 等），而 1024~65535 的端口号是动态分配的，用于标识用户级别的网络应用程序。

总之，端口是网络通信中的一个非常基础、重要的概念，用于标识网络应用程序或服务，位于 TCP/IP 七层模型的传输层。

Last-modified返回的是什么？

Last-Modified 是 HTTP 协议中的一个响应头部字段，会在服务器响应客户端请求时携带一个表示资源最后修改时间的时间戳。具体来说，该字段表示的是服务器上请求的资源最后一次修改的时间，通常格式是 GMT 格式的时间戳。

当客户端请求一个资源时，如果该资源有 Last-Modified 响应头，则客户端会记录下该时间戳。在之后客户端再次请求该资源时，会在请求中加入一个 If-Modified-Since 头部字段，其为上次请求中记录的 Last-Modified 值。服务器接收到请求后，会将请求头中的 If-Modified-Since 值与服务器上该资源的最后修改时间比较。如果资源的最后修改时间大于 If-Modified-Since，说明资源在此期间发生了修改，服务器会重新返回该资源及最新的 Last-Modified 值；反之，如果资源的最后修改时间不大于 If-Modified-Since，说明资源未发生修改，服务器会返回 304 Not Modified 状态码，告诉客户端可以使用本地缓存而无须重新下载该资源。

因此，Last-Modified 的作用是协助客户端实现缓存控制，避免重复请求和数据传输，同时还能降低服务器端的负载。

JSON.parse / JSON.stringify有什么弊端？

JSON.parse() 和 JSON.stringify() 是在 JavaScript 中用来处理 JSON 格式数据的两个方法。

它们的弊端主要有以下几点：

  1. 安全问题：JSON.parse() 可能会导致代码注入。如果将一个包含恶意代码的 JSON 字符串传入 JSON.parse() 中，就会执行该恶意代码，可能导致安全问题。
  2. 限制：JSON.parse() 和 JSON.stringify() 都有一定的局限性。比如 JSON.parse() 无法处理循环引用的对象，JSON.stringify() 不能序列化函数和 RegExp 等类型的数据。
  3. 性能：在处理大型 JSON 数据时，JSON.parse() 和 JSON.stringify() 的性能可能会较慢，会占用大量的 CPU 和内存。

为了解决这些问题，可以使用第三方的库或框架，如 jQuery 的 $.parseJSON() 和 JSON2.js 等，它们提供了更安全、更全面和更高效的 JSON 数据处理方案。同时，在使用 JSON.parse() 或 JSON.stringify() 时，也需要遵循相关的安全规范和最佳实践，以减少潜在的风险和问题。

循环引用有什么解决办法？

循环引用是指两个或多个对象互相引用，从而形成一个闭环。JSON.stringify() 方法默认无法处理循环引用，当序列化包含循环引用的对象时，会抛出 TypeError 异常。在 JavaScript 中，循环引用通常是由于对象间互相引用造成的。

针对循环引用的解决办法如下：

  1. 删除循环引用：在对象中删除循环引用属性，避免出现循环引用问题。例如，可以通过手动删除循环引用属性或使用对象浅拷贝的方式来解决。
  2. 自定义序列化：自定义一个函数来序列化对象，将循环引用对象标记为 null 或其他特定的值，以避免抛出 TypeError 异常。例如，可以使用递归的方式来遍历对象，判断是否有循环引用，并在有循环引用时返回一个 null。
  3. 序列化和反序列化时保存引用关系：使用第三方库来支持序列化和反序列化时保存引用关系。例如，可以使用 CircularJSON 库来支持 JSON 序列化和反序列化。CircularJSON 库支持循环引用，可以将包含循环引用的对象序列化成 JSON 字符串，然后解析成包含循环引用的对象。

总之，在处理循环引用对象时需要特别注意，在保证序列化和反序列化的正确性的情况下，采取合适的解决办法避免出现异常情况。

setTimeout能到准确的时间就执行回调吗？为什么？

setTimeout并不能保证在准确的时间执行回调函数。setTimeout的作用是在指定的时间之后将代码插入到事件循环队列的末尾，等待当前任务执行完毕后执行定时器中的回调函数。但是，由于JavaScript是单线程执行的，如果在执行回调函数之前其他任务正在执行，或者JS引擎正在繁忙地处理其他代码，那么setTimeout的回调函数就会被推迟执行。

另外，浏览器的事件循环机制也会影响setTimeout的执行时间。如果事件队列中已经有其他任务在等待被执行，那么setTimeout的回调函数也会被延迟执行。具体来说，如果在当前宏任务中执行了很多耗时的任务，那么事件队列中的下一个任务（即setTimeout）就会被放置到时间等待列表的末尾，等待下一个时间片再被执行。

因此，setTimeout并不能够保证在准确的时间点执行回调函数，它只是一种在未来指定时间后异步执行代码的机制。如果需要在更精确的时间点执行代码，可以使用Web Worker等其他机制。

现在的主流框架，是怎么实现响应式布局设计的？

现在的主流框架（如Bootstrap、Material Design、Foundation等）一般都采用弹性网格系统（Flexible Grid System）和媒体查询（Media Queries）的方式实现响应式布局设计。

弹性网格系统允许设计师定义一个网格结构，并将网格分成若干列和固定的宽度，而网格的高度则根据所含内容的多少而自适应变化。通过按比例分配屏幕宽度中的百分比，布局可以随着程序的运行而动态调整。

媒体查询是CSS3中的一个很有用的功能，能够根据设备屏幕大小、分辨率等特性，在不同的情况下应用不同的样式，使页面能够在不同的设备上得到最佳的展示效果。

通过使用弹性网格和媒体查询的方式，主流框架实现了响应式的布局设计。在实际开发中，开发者只需要按照预设的网格系统来布局，同时利用媒体查询来设置不同的CSS样式，就可以让Web页面适配不同的设备。

什么是React全量更新？

React 应用在进行渲染时，会根据组件的变化情况选择性地更新组件。在 React 中，全量更新（Full DOM Update）就是指对整个 DOM 树进行重新渲染，即从根节点开始，按照树形结构逐层更新每个节点，并重新渲染所有子组件。

全量更新可能会带来性能问题，会导致页面重新渲染的速度变慢，即使那些暂时没有变化的部分也被重新渲染了。这是因为全量更新是一种暴力的方式，React 需要重新计算和比较每一个组件的状态和属性，无法区分哪些组件真正需要更新，哪些不需要更新。

因此，React 引入了 Virtual DOM，它是 DOM 的轻量级副本。当组件状态或属性发生变化时，React 先在 Virtual DOM 中更新，并且比较新旧 Virtual DOM 的差异，只对差异部分进行更新，最终只更新需要更新的部分，从而避免了全量更新。

在一些特殊情况下，全量更新是必要的，例如：

  1. 首次渲染整个页面。
  2. 当应用的状态发生严重错误时，需要强制重新渲染整个应用。
  3. 当组件之间的关系发生变化，无法准确计算出哪些组件需要更新时。

但一般情况下，为了提高 React 应用的性能，应该尽量避免全量更新，而是选择使用 Virtual DOM 进行部分更新。

React 全量更新会带来性能损耗，这个问题是怎么解决的？

React 组件的全量更新确实会带来性能问题，尤其是在组件树比较大时，会消耗相当多的计算资源，导致页面卡顿、动画不流畅等问题。

针对这个问题，React 采取了一系列优化措施，包括：

  1. shouldComponentUpdate 方法：该方法用于在组件进行更新之前，判断组件的 props 和 state 是否发生变化，如果没有变化则会阻止组件的更新。
  2. PureComponent 组件：该组件内部实现 shouldComponentUpdate 方法的浅比较逻辑，从而减少全量更新的次数。
  3. React.memo 高阶组件：类似于 PureComponent，但用于函数式组件，可以缓存组件的结果，从而减少重复渲染。
  4. 使用 key 属性：key 属性可以帮助 React 识别具有相同结构的不同组件，并根据需要进行复用或重渲染。

通过以上优化措施，可以减少 React 组件的全量更新，从而提高组件的性能和响应速度。

React是通过什么机制来判断当前节点是旧的，并需要更新的呢？

在 React 中，每个组件都有一个虚拟 DOM 树，当组件的 props 或 state 发生变化时，React 会使用组件的 render 方法重新创建虚拟 DOM 树。然后，React 会使用 Diff 算法比较旧的虚拟 DOM 树与新的虚拟 DOM 树，以便确定哪些节点需要被更新。

Diff 算法是 React 实现组件更新和 DOM 操作的核心算法。它比较两棵虚拟 DOM 树，找出两者之间的区别，并只更新必要的 DOM 节点。在 Diff 算法中，React 会遍历两棵虚拟 DOM 树，比较每个节点的类型和属性，并确定节点是被删除、添加、替换，还是仅需要更新。根据 Diff 算法的结果，React 会创建出一系列更新操作，用于更新真实 DOM。

在进行 Diff 算法时，React 会为每个虚拟 DOM 树中的节点添加一个 key 属性，以便更准确地比较它们。如果一组节点没有 key 属性，那么 React 会比较它们的位置和属性。但是，如果两个节点的位置发生了变化，那么 React 将会删除旧的节点，并将新的节点添加进来，这会导致重复创建 DOM 节点和大量的不必要的 DOM 操作，从而影响了应用的性能。

综上所述，React 使用虚拟 DOM 和 Diff 算法来判断当前节点是否需要更新，并通过优化比较算法和 key 属性来提高组件的更新效率。

ESLint、Prettier、Husky、Lint-Staged分别干什么用？

  • ESLint 是一个非常流行的代码检查工具，可以捕捉代码中的潜在问题，例如未声明的变量、不兼容的语法特性、格式等，并给出适当的警告或错误。ESLint 可以在代码编写期间（如存储库的推送前）或维护期间（如 PR 检查时）以指定的规则集运行。ESLint 有许多配置和插件，可以使其适应各种项目需求。

  • Husky 是一个 Git 钩子的管理工具，可以在特定事件（例如代码提交前）和 Git 命令之前或之后运行指定的脚本。 Husky 用于在提交代码之前运行本地 linting 和测试（以便在 CI 阶段前快速捕获问题或出现错误）。
  • Lint-Staged 在 Git commit 过程中且仅在您准备提交的特定文件中运行指定的脚本。它类似于 Husky，但它更为专注于单个提交。这使得例如在提交时仅对目录下即将修改的文件运行 lint 变得非常容易。

红黑树的概念及应用场景？

红黑树是一种自平衡的二叉搜索树，通过在插入和删除的过程中进行旋转和颜色变换，来保持树的平衡。

红黑树的特点如下：

  1. 每个节点都有颜色，红色或黑色。
  2. 根节点是黑色的。
  3. 叶子节点都是黑色的空节点。
  4. 如果一个节点是红色的，则它的两个子节点都是黑色的。
  5. 从任意一个节点到叶子节点的路径，经过的黑色节点数是相同的。

红黑树的应用场景包括：

  1. C++ STL 的 set 和 map 实现都是基于红黑树的，因为红黑树的查找、插入、删除都可以在 O(log n) 的时间内完成，同时它还能保持自平衡，防止退化为链表。
  2. Linux 内核中的进程调度器 Completely Fair Scheduler 也是基于红黑树实现的，通过红黑树来维护进程的优先级顺序。
  3. 数据库索引的 B+ 树等也采用了红黑树的思想。

总之，红黑树是一种十分重要且常用的数据结构，适用于需要高效地进行增查删操作且需要保持平衡的场景。

type和interface的区别？

在TypeScript中，type和interface都可以用于定义类型，但是它们有一些不同的特性和用途。

首先，interface只能定义对象类型，而type可以定义各种类型，例如原始类型、联合类型、元组类型和函数类型等，因此type比interface更加灵活。

其次，interface支持扩展和继承，可以通过extends关键字来扩展其他interface的定义，而type不支持直接继承和扩展其他类型的定义。

此外，使用interface可以更自然描述对象的形状（属性数量、属性类型），而type更多用于联合、交叉、映射等操作，或对已有类型进行修改（比如将所有属性设置为可选或只读）。

最后，在实际使用中，二者可以互换使用，但是在一些场景下，使用type会更加方便和直观，而在其他场景下，使用interface则更加适合。

const和readonly的区别？

在 TypeScript 中，const 和 readonly 都用于创建只读变量，但它们的用法和含义略有不同。

const 声明的变量是常量，其值不能被重新分配。这意味着一旦变量被定义并初始化为某个值，就无法再更改该值，当你尝试改变一个 const 变量时，TypeScript 编译器会报错。

readonly 声明的变量也是只读的，但是它们是非常灵活的，可以在变量的声明和实例化过程中使用。readonly 变量的值只能在变量创建时或构造函数中被分配一次。这样，类中的成员变量可以被声明为readonly，并且只能在初始化时或构造函数中被分配。

两者之间的主要区别是 const 变量必须在声明时进行初始化，而 readonly 变量可以在类构造函数中初始化。所以如果你需要创建只读类成员时，使用readonly 关键字，而 const 变量则更适合用在其他上下文中，例如函数作用域。

any、unknown、never的区别？

在 TypeScript 中，any、unknown、never 都是基础类型。

  • any 类型表示任何类型都可以，它可以被用来逃避 TypeScript 类型检查。将一个变量声明为 any 类型之后，在编译时这个变量的类型检查会被关闭，因此该变量可以接受任何类型的值。 当一个变量的类型未知时，你可以使用 any，但是为了避免运行时错误，尽可能使用更具体的类型。

  • unknown 类型表示未知类型。与 any 类型不同的是，使用 unknown 类型后类型检查并不会被放松，可以被强制转换为其他类型。使用 unknown 类型可以确保代码的类型检查健壮性。

  • never 表示一个永远不会有返回值的类型，它主要用于描述那些抛出异常或死循环的函数，或者是那些永远不会结束的函数。

总之，any 类型是 TypeScript 的一项灵活性，它允许你创建允许任意类型的变量，但同时也意味着该变量不受静态类型检查的保护；unknown 类型则是一个更安全的替代方案，它允许你控制对未知类型的使用，并在必要时进行强制类型转换；never 类型表示一个永远不会有返回值的类型，它主要用于描述那些不正常结束的函数。

内置类型，如Omit、Pick、Parameter等的作用和区别？

在 TypeScript 中，Omit、Pick、Parameter 等都是内置类型，以下是它们的作用和区别：

  • Omit<T, K>：从类型 T 中删除属性 K，返回新的类型。在某些场景下，可以使用 Omit 简化原始类型，例如用于声明应用程序状态，其中一些属性已被其他属性所表示，从而减少状态的复杂度。

  • Pick<T, K>：从类型 T 中提取 K 属性，返回新的类型。在某些场景下，可以使用 Pick 从原始类型中选择部分属性，例如在定义 React Props 时，从 interface 中取出部分属性。

  • Parameter<T>：获取函数类型 T 的参数类型组成的元组类型。在编写通用函数时，可以使用 Parameter 获取函数参数的类型信息。

总之，Omit、Pick、Parameter 等内置类型可以帮助我们尽可能简洁地定义类型，从而提高我们的代码可读性，简化代码编写的复杂性，避免一些潜在的类型匹配错误和运行时类型错误。

tsconfig中一般需要进行哪些配置？

以下是几个常用的 tsconfig.json 配置选项：

  1. compilerOptions.target: 指定编译的目标类型，可选值为 ES3、ES5、ES6/ES2015、ES2016、ES2017、ES2018、ES2019、ES2020、ESNext，以及各种浏览器支持的版本号。一般建议选择较新的版本。
  2. compilerOptions.module: 指定用哪种模块格式进行输出，可选值为 CommonJS、AMD、System、UMD、ES6/ES2015、ES2020、ESNext 等。一般建议选择 ES6/ES2015 或更高的版本，以支持 Tree-shaking 和其他 ES6+ 特性。
  3. compilerOptions.lib: 指定 TypeScript 编译器可以使用的库文件，常用选项为 es5, es6, es7, es2015.promise, es2015.collection 等。如果没有指定，则默认使用所有支持的库文件。
  4. compilerOptions.outDir: 指定编译后的 JavaScript 文件存放的目录。
  5. compilerOptions.sourceMap: 是否生成 source map 文件。如果启用，将会在编译后生成与 TypeScript 代码对应的 JavaScript 代码的映射文件，方便调试。
  6. compilerOptions.allowJs: 是否允许编译器将 JavaScript 文件作为输入，可以为 true 或 false。
  7. compilerOptions.strict: TypeScript 的“严格模式”，包含了一些额外的检查，如禁用隐式 any 类型、强制 Function 必须有返回值、强制对象属性必须初始化、禁止对 null/undefined 取属性、严格限制类型相等等。建议启用。
  8. compilerOptions.noUnused*: 是否检查未使用的变量和导入，例如 noUnusedLocals, noUnusedParameters, noUnusedImports`。建议启用以优化代码质量。
  9. compilerOptions.eslint*: 是否将 TypeScript 编译器与 ESLint 集成，例如 eslintPlugin, eslintPath, eslintFilePatterns` 等。可以在开发过程中进行类型和语法的检查，提高代码质量。

还有其他一些选项，根据项目需要进行选择和配置。
